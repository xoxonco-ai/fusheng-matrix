# 資料層：AI 能讀什麼

## 這個任務的資料跟一般知識庫不一樣

浮生矩陣的報告生成 AI**不讀靜態知識庫**，它只讀兩份「當次訂單」的動態輸入：

| 欄位 | 內容 | 來源 |
|---|---|---|
| `order.summary` | 命盤摘要（八字/紫微/占星/人類圖四套系統的計算結果文字） | 由 `paipan.html`／`hepan.html` 的排盤引擎算出，下單時存進 `orders` 表 |
| `order.evidence` | 使用者自己提供的真實事件（選填，作為報告裡的「鐵證」段落） | 使用者下單表單填寫 |
| `order.intensity` | 語氣力道（溫和／犀利／狠） | 使用者下單選擇，對應 [rules/tone-map.md](../rules/tone-map.md) |
| `order.birth.relation` | 合盤版的關係階段（曖昧探索／熱戀磨合／穩定婚姻／修復期） | 使用者下單選擇（僅合盤版） |

這四項就是 `generate-report.ts` 唯一會讀到的「資料」，沒有額外的 FAQ/案例/公司資料檔案。

## 排盤計算常數不算這一層

`paipan.html`／`hepan.html` 裡的八字天干地支對照、人類圖閘門/通道/類型等表格，是**排盤引擎的計算常數**（給程式算命盤用），不是給 AI 寫報告讀的知識資料——AI 只看排盤算完之後產出的摘要文字。這兩者容易搞混，這裡特別註明：**排盤計算常數不需要、也不應該搬進 `knowledge/data/`**（先前版本的文件把這兩者混為一談，這裡已修正）。

## 如果未來要擴充

如果之後想讓 AI 多讀一些「案例庫」（例如過往報告裡寫得特別準的段落，當作 few-shot 範例），可以放在這裡，例如：

- `case-studies/` 過往報告案例（去識別化後）
- `faq/` 常見客戶售前問題

目前這兩個資料夾都還不存在，也沒有實際檔案。

### 格式建議

- 一律使用 Markdown
- 只放真實資料，不放測試假資料（若測試需要，應統一存放在專屬的測試目錄如 `knowledge/data/fixtures/` 中）
